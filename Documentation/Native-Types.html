Native Types

Spin comes with a variety of basic types ready to store information in memory.
Those types are:

Type Size Description
Byte 1 Byte Unsigned Integer
Natural 8 Bytes Unsigned Integer
Integer 8 Bytes Signed Integer
Boolean 1 Byte Boolean Value
Character 1 Byte Character
Real 8 Bytes Real Number
Imaginary 8 Bytes Imaginary Number
Complex 16 Bytes Complex Number
String 8N Bytes ASCII String

Types with a size exceeding 8 bytes need a pointer and can't possibly live on the stack, so there's a small 8 bytes overhead in addition to the data.

Types with that characteristic, such as Complex or String are defined "virtual", meaning that they can be modified by their own methods while normal basic types live on the stack and can never be modified by methods of their own because mutation over classes requires a pointer to the current instance. Those procedures that mutate the intance can be chained with the infix chain operator -> which will force the virtual machine to leave a copy of the instance pointer on top of the stack instead of consuming it.

Arrays and Lists

Since Spin is a multidimensional language, the syntax allows arrays with the following type declaration:

Production Rule

array type → '[',type,']';

Spin arrays are dynamic arrays, contiguous in heap memory, that automatically shrink or expand when needed.

Lamdas and Function Types

Spin allows lamdas, functional structures that can be passed as parameters or stored in variables.
The lamda type follows the grammar:

Production Rule

lamda type → 'ƒ','(',(type|(',',type)*),')',(':',type)?;

Type Node

Types are usually stored in trees-like data structures because of their underlying complexity.
The final type grammar follows:

Production Rule

type → basic type|array type|lamda type;

Type Annotation

While working with Spin, whenever a type annotation is required, we follow the expression with a tag. A tag is made of a colon : followed by a type. The grammar follows:

Production Rule

tag → ':',type;
